cron entry. Exact logic:

```sh
#!/usr/bin/env sh
# install-cron.sh — idempotently install an hourly cron entry for job-board-aggregator.
# Usage:  sh scripts/install-cron.sh
#         npm run schedule:install
#
# Installs: 0 * * * *  cd <projdir> && node scan.mjs --notify >> <projdir>/scan.log 2>&1
# Guard:    does nothing if this project's scan.mjs entry already exists in crontab.

set -e

PROJ_DIR="$(cd "$(dirname "$0")/.." && pwd)"
CRON_CMD="0 * * * * cd $PROJ_DIR && node scan.mjs --notify >> $PROJ_DIR/scan.log 2>&1"

# Idempotency guard: exit cleanly if already installed
if crontab -l 2>/dev/null | grep -qF "cd $PROJ_DIR && node scan.mjs"; then
  printf 'Cron entry already installed for %s. Nothing to do.\n' "$PROJ_DIR"
  exit 0
fi

# Append to existing crontab (preserves other entries)
{ crontab -l 2>/dev/null; printf '%s\n' "$CRON_CMD"; } | crontab -

printf 'Installed cron entry:\n  %s\n' "$CRON_CMD"
printf 'Verify with: crontab -l\n'
printf 'Remove with: crontab -e  (delete the line manually)\n'
```

Create `.github/workflows/scan.yml`. The directory `.github/workflows/` likely does not
exist — create it. The workflow must:

1. Trigger on `schedule:` cron `'0 * * * *'` (hourly) AND `workflow_dispatch:` (manual test)
2. Use `actions/cache@v4` to RESTORE `data/jobs.db` BEFORE the scan step
   - Cache key: `jobs-db-v1-${{ github.run_id }}` (unique per run → always a save)
   - restore-keys: `jobs-db-v1-` (prefix match picks up most recent saved DB)
3. `mkdir -p data` before the scan (directory may not exist in a fresh checkout)
4. Run `node scan.mjs --notify` with `DISCORD_WEBHOOK_URL: ${{ secrets.DISCORD_WEBHOOK_URL }}`
5. The actions/cache post step automatically saves `data/jobs.db` after the job with key
   `jobs-db-v1-${{ github.run_id }}` — no explicit save step needed since the unique key
   per run guarantees a cache miss → auto-save on post.

Exact file content:

```yaml
name: Job Board Scan

on:
  schedule:
    - cron: '0 * * * *'  # every hour
  workflow_dispatch:       # manual trigger for testing

jobs:
  scan:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '22'

      - name: Restore dedup DB
        uses: actions/cache@v4
        with:
          path: data/jobs.db
          key: jobs-db-v1-${{ github.run_id }}
          restore-keys: |
            jobs-db-v1-

      - name: Ensure data directory exists
        run: mkdir -p data

      - name: Run scan
        env:
          DISCORD_WEBHOOK_URL: ${{ secrets.DISCORD_WEBHOOK_URL }}
        run: node scan.mjs --notify
```

Note on cache strategy: `restore-keys: jobs-db-v1-` picks up the last run's DB snapshot
(partial key match). The `key: jobs-db-v1-${{ github.run_id }}` is always unique so it is
always a cache miss → actions/cache post step saves the updated DB automatically. This
creates a rolling chain of DB snapshots ensuring each run is incremental.
