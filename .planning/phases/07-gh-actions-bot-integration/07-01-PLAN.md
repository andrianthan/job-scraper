# Plan 07-01: GH Actions bot pushes CSV to jobs-data

## Objective

Build the CI-side plumbing that takes the CSV produced by `csv-writer.mjs`
(Phase 6) and commits/pushes it to the separate `andrianthan/jobs-data` repo
on each scan run. Deliverables: a `scripts/push-csv.mjs` helper, extended
`.github/workflows/scan.yml`, updated `.env.example`, and unit tests covering
the helper's command construction + error paths.

## Context Recap

Phase 6 built the writer. Phase 7 ships the bot. Phase 8 will splice
`appendRows()` into `scan.mjs --notify` and pin a Discord message with the
raw URL. Phase 7's helper reads new jobs from a JSON file the workflow
captures (via `scan.mjs --json`), runs `appendRows()` against the jobs-data
checkout, then runs `git add/commit/push`. Operator config: `GH_TOKEN` secret
in this repo (PAT with `contents:write` on `andrianthan/jobs-data` only).

13 locked decisions in `07-CONTEXT.md`; key ones referenced per task.

<action>
Create `scripts/push-csv.mjs` (zero npm deps, ESM, `node:child_process.spawn` + `node:fs/promises` + `node:path`).

Public API: `export async function pushCsv({ jobs, env }) -> { ok, rowsAppended, totalRows, committed, pushed, error? }`. The `env` param is a `process.env`-like object (read keys: `GH_TOKEN`, `JOBS_DATA_REPO`, `CSV_PATH`, `GITHUB_RUN_URL`); defaults to `process.env` when omitted. `jobs` is `Array<{url, company, title, location, source, postedAt?, fallbackUrl?}>`.

Behavior:
1. Resolve `JOBS_DATA_REPO` = `env.JOBS_DATA_REPO || 'andrianthan/jobs-data'`. If not set at all, log `CSV push: JOBS_DATA_REPO not set, defaulting to andrianthan/jobs-data` to stderr. (Operator can override for forks.)
2. Check `env.GH_TOKEN`. If missing/empty, exit with `{ ok: false, error: 'GH_TOKEN not set -- skipping CSV push (this is expected for local dev)' }` BEFORE any git call. CLI mode: `process.exitCode = 1`, print the same string to stderr. (D-09: fail loud, no silent success.)
3. Resolve `CSV_PATH` = `env.CSV_PATH`. CLI mode default = `./jobs-data/jobs.csv` (relative to cwd, where the workflow cd's before running).
4. Call `appendRows(jobs, { filePath: CSV_PATH })` from `./csv-writer.mjs`. Catch any error and return `{ ok: false, error: \`appendRows failed: \${err.message}\` }` with exit 1.
5. If `rowsAppended === 0` and file existed before, log `CSV push: 0 new jobs to append` to stderr and skip remaining steps; return `{ ok: true, rowsAppended: 0, totalRows, committed: false, pushed: false }` with exit 0. (D-06: no-change skip.)
6. If file did not exist before (cold start), proceed to step 7 even if `rowsAppended === 0` (writer still wrote header + separator).
7. Run `git diff --quiet HEAD -- jobs.csv` (or the relative path) via `spawnSync('git', ['diff', '--quiet', 'HEAD', '--', 'jobs.csv'], { cwd: jobsDataDir, stdio: 'pipe' })`. Use `process.cwd()` as `jobsDataDir` unless `CSV_PATH` is absolute. If exit code 0 → log `CSV push: no changes, skipping` to stderr; return `{ ok: true, rowsAppended, totalRows, committed: false, pushed: false }` with exit 0. (D-06 belt-and-suspenders; writer-level dedup is the first line of defense.)
8. Set `git config user.name "github-actions[bot]"` and `user.email "41898282+github-actions[bot]@users.noreply.github.com"` in `jobsDataDir`. (D-04: GitHub Actions bot identity.)
9. `git add jobs.csv` (or basename of `CSV_PATH` if different). Spawn with `stdio: 'inherit'` so errors surface.
10. Build commit message: `chore(csv): append \${rowsAppended} new jobs [skip ci]\n\nTriggered by: \${env.GITHUB_RUN_URL || 'local run'}`. (D-05: includes run URL for auditability; `[skip ci]` prevents loop if jobs-data ever gets a workflow.)
11. `git commit -m <message>` (use `spawnSync` with `input` piped via `-F -` and a temp file or array form: `git commit -m "<line1>" -m "<line2>"`).
12. `git push https://x-access-token:\${env.GH_TOKEN}@github.com/\${JOBS_DATA_REPO}.git HEAD:main`. (D-03: PAT in URL.)
13. On any spawn error or non-zero exit, return `{ ok: false, error: <message>, rowsAppended, totalRows, committed: false, pushed: false }` and CLI mode sets `process.exitCode = 1`.
14. On success, return `{ ok: true, rowsAppended, totalRows, committed: true, pushed: true }`.

CLI mode (when run as `node scripts/push-csv.mjs <path-to-jobs.json>`):
- Read JSON file from argv[2]. Parse to `jobs` array. If argv[2] missing, print usage to stderr and exit 1.
- Call `pushCsv({ jobs, env: process.env })`.
- Set `process.exitCode = result.ok ? 0 : 1`.
- On success, print `CSV push: \${result.rowsAppended} new rows, total \${result.totalRows}, pushed=\${result.pushed}` to stderr.

Top of file: `#!/usr/bin/env node` shebang (harmless in ESM). Comment block citing Phase 7 D-01..D-13 for traceability.

Wrap push URL token: use URL-encoding for the GH_TOKEN value to handle special chars (`encodeURIComponent`). Spawn args must be array form (no shell) to avoid injection.

## Notes

- **Why `if: always()` on capture/push steps:** the scan step can fail (network, bad boards); we still want the CSV to reflect whatever jobs DID get discovered before failure, and the workflow to commit a partial update rather than lose it.
- **Why `persist-credentials: false` on the jobs-data checkout:** the helper uses its own `git push` with the PAT, not the checkout's credentials. Disabling persist prevents the runner from caching credentials it doesn't need.
- **No new job, no matrix:** one job keeps the YAML short and the failure modes simple. If the helper grows beyond ~150 lines, consider splitting into its own workflow triggered by `workflow_run`.
- **Cold-start race:** the writer's atomic-rename pattern (D-03 in Phase 6) means a partial write can never leave a corrupted `jobs.csv`. If the runner dies between `writeFile` and `rename`, the next run starts fresh — the next `appendRows` call reads ENOENT and writes header + rows again.
